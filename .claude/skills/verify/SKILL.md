# Проверка Stream Director на живом приложении

## Рецепт (проверен на LoL-пайплайне)

1. **Изолированный конфиг**: запускать приложение с cwd = временная папка —
   `DATA_DIR = Path("data")` относительный, поэтому свежий `data/settings.json`
   не трогает реальный. Полезные оверрайды:
   `{"lol_url": "http://127.0.0.1:<порт>", "server_port": 8765,
   "global_cooldown_s": 1.0, "debounce_s": 0.3, "voice_enabled": false}`.
   Без LLM-ключа реплики идут по фолбэк-шаблонам — детерминированный путь.
2. **Фейковый источник LoL**: обычный HTTP-сервер с эндпоинтом
   `/liveclientdata/allgamedata`, отдающий снапшот, меняющийся по времени
   (поллер ходит раз в секунду; https не обязателен — url настраивается).
   Формат снапшота — см. `tests/test_lol_mapper.py::team_payload`.
3. **Запуск**: `python -m stream_director` (пакет установлен editable).
4. **Наблюдение**: реплики выходят в `ws://127.0.0.1:<server_port>/ws/overlay`
   как JSON `{"type": "replica", "text": ..., "effect": <stimulus.type>}`.
   Клиент — питоновский `websockets` (установлен как зависимость uvicorn).

## Грабли

- **stdout приложения обязан читаться или идти в файл**: httpx логирует каждую
  секунду; заполненный PIPE без читателя блокирует logging и морозит ВСЁ
  приложение (события перестают обрабатываться).
- На старте ~20–30 с приложение может не отвечать на WS-пинги (грузится
  Silero TTS) — у ws-клиента ставить `ping_interval=None` или переподключаться.
- Консоль Windows отдаёт кракозябры — ставить `PYTHONIOENCODING=utf-8`.
- Тихие стимулы (`team_state`, `low_hp`, `battle_result` LoL) в overlay НЕ
  приходят — их отсутствие в WS это норма, проверять по эффекту на память.
- Союзные события LoL идут не чаще одного в 60 с (`_ALLY_EVENT_INTERVAL_S`) —
  сценарий проверки растягивать по времени, а не слать всё сразу.
