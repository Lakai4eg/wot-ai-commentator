# Проверка Stream Director на живом приложении

## Рецепт (проверен на LoL-пайплайне)

1. **Изолированный конфиг**: запускать приложение с cwd = временная папка —
   `DATA_DIR = Path("data")` относительный, поэтому свежий `data/settings.json`
   не трогает реальный. Полезные оверрайды:

   ```json
   {"lol_url": "http://127.0.0.1:<порт фейкового LoL>", "server_port": 8765,
    "llm_provider": "openai", "openai_base_url": "http://127.0.0.1:<порт фейкового LLM>/v1",
    "openai_model": "fake", "debounce_window_s": 1.0, "global_cooldown_s": 1.0,
    "voice_enabled": false}
   ```

2. **Фейковый LLM**: локальный HTTP-сервер с `POST /v1/chat/completions`,
   отвечающий фиксированным текстом в формате OpenAI
   (`{"choices": [{"message": {"content": "..."}}]}`) и **логирующий каждый
   полученный промпт** в файл. Промпт-лог — главный инструмент проверки: по нему
   видно, что именно ушло в LLM (бриф, склейка, `!dir`). Шаблонов реплик в
   проекте нет: без живой LLM комментатор молчит, поэтому фейковый эндпоинт
   обязателен.
3. **Фейковый источник LoL**: обычный HTTP-сервер с эндпоинтом
   `/liveclientdata/allgamedata`, отдающий снапшот, меняющийся по времени
   (поллер ходит раз в секунду; https не обязателен — url настраивается).
   Формат снапшота — см. записи реальных матчей в `data/lol-events/*.jsonl`
   и разбор полей в `games/lol/mapper.py`.
4. **Фейковый чат** (нужен для `!dir`): `TwitchChatReader` ходит на
   `HOST`/`PORT` — модульные константы `chat/twitch.py`. Поднять локальный
   IRC-сервер (принять `NICK`/`JOIN`, слать
   `:nick!nick@nick.tmi.twitch.tv PRIVMSG #канал :!dir текст`) и запускать
   приложение обёрткой, которая переставляет константы на него:
   `python -c "import stream_director.chat.twitch as t; t.HOST='127.0.0.1'; t.PORT=<порт>; from stream_director.main import main; main()"`.
   Ник заказчика должен лежать в белом списке (`POST /api/users`) или включить
   `commands_open_to_all`.
5. **Запуск**: `python -m stream_director` (пакет установлен editable).
6. **Наблюдение**: реплики выходят в `ws://127.0.0.1:<server_port>/ws/overlay`
   как JSON `{"type": "replica", "text": ..., "effect": <stimulus.type>}`.
   Клиент — питоновский `websockets` (установлен как зависимость uvicorn).

## Что проверяем

- **Бриф**: в промпт-логе фейкового LLM есть мета-промпт генерации брифа (слово
  «БРИФ»), а в `data/chat-users.db` — строка в `game_briefs`
  (`SELECT game, subject FROM game_briefs`). Триггер — `battle_start`.
- **Склейка**: три события подряд в пределах окна дают ОДИН промпт с блоком
  «Что произошло за последние … с» и ОДНУ реплику в оверлее.
- **Заказ `!dir`**: даёт отдельную реплику, а в его промпте есть тег `<заказ>`
  и НЕТ блока склейки — заказ идёт мимо окна.

## Грабли

- **stdout приложения обязан читаться или идти в файл**: httpx логирует каждую
  секунду; заполненный PIPE без читателя блокирует logging и морозит ВСЁ
  приложение (события перестают обрабатываться).
- На старте ~20–30 с приложение может не отвечать на WS-пинги (грузится
  Silero TTS) — у ws-клиента ставить `ping_interval=None` или переподключаться.
- Консоль Windows отдаёт кракозябры — ставить `PYTHONIOENCODING=utf-8`.
- Тихие стимулы (`team_state`, `low_hp`, `battle_result` LoL) в overlay НЕ
  приходят — их отсутствие в WS это норма, проверять по эффекту на память.
- Союзные события LoL идут не чаще одного в 60 с (`_ALLY_EVENT_INTERVAL_S`) —
  сценарий проверки растягивать по времени, а не слать всё сразу.
- Реплика уходит по главному событию пачки, `effect` = его `stimulus.type`:
  ждать в оверлее по одному эффекту на каждое событие бессмысленно.
